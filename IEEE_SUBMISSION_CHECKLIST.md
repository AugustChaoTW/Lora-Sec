- [ ] **協議與形式化模型 (Protocol & Model)**

- [ ] **修補驗證 (Patch Verification)**

  - [ ] 基線 vs 修補的對比數據
  - [ ] 表格與圖表（攻擊成功率、延遲、能耗）

- [ ] **相關工作 (Related Work)**

  - [ ] 無線路由安全（4-5 篇）

### 第三層：評審準備 (REVIEWER PREPARATION)

#### 🎯 常見質疑與預回應

**質疑 1: "模型是否過度簡化？"**
- 預防對策：
  - [ ] 清晰說明每個抽象的合理性
  - [ ] 表格展示「抽象 vs 實現」的映射
  - [ ] 論證「簡化下的漏洞 = 實現中的漏洞」
  - [ ] 提供敏感性分析（鬆綁某些假設後結果穩定性）

**質疑 2: "攻擊者模型是否太強？"**
- 預防對策：
  - [ ] Dolev-Yao 模型的標準應用論證
  - [ ] 表格展示「攻擊者能力」vs「現實威脅」的對應
  - [ ] 強調：「協議安全即使在強攻擊者下也應達成」
  - [ ] 提供折減攻擊者能力下的驗證結果（optional）

**質疑 3: "修補成本評估是否完整？"**
- 預防對策：
  - [ ] 表格列舉所有開銷（訊息大小、CPU、能耗、延遲）
  - [ ] 數據源明確標注（實測 vs 計算 vs 估計）
  - [ ] ns-3 實驗數據支持
  - [ ] 與其他協議（如 SEAD）的對比

**質疑 4: "實驗規模是否足夠？"**
- 預防對策：
  - [ ] ns-3 實驗涵蓋多種拓撲（星形、樹形、環形）
  - [ ] 节點數涵蓋 10-100（小規模 → 中規模）
  - [ ] 攻擊場景涵蓋 3-5 種（metric spoofing, relay, etc.）
  - [ ] 統計意義（重複 10 次、報告均值 ± std）

**質疑 5: "可重現性如何保證？"**
- 預防對策：
  - [ ] GitHub 公開代碼（Tamarin 模型 + ns-3 代碼）
  - [ ] Docker 容器化環境
  - [ ] 詳細 README（安裝、運行、驗證步驟）
  - [ ] 預期輸出截圖與對比

## 最後提醒

### 🚩 常見失敗原因（避免）

1. ✗ 形式化驗證結果與論文敘述不一致
   - 預防：Tamarin 反例 → 論文攻擊 mapping 表

2. ✗ 修補設計未充分驗證
   - 預防：確保修補模型所有 5 個 lemma PROVED

3. ✗ 攻擊威脅程度誇大
   - 預防：基於實驗數據定量說明，避免「可能」、「可能導致」

4. ✗ 實驗缺乏統計意義
   - 預防：每個場景重複 ≥10 次，報告均值 ± 標準差

5. ✗ 論文複製貼貼教科書內容
   - 預防：所有敘述都應指向本工作的新發現

6. ✗ 忽視可重現性
   - 預防：代碼公開 + Docker 容器 + 詳細 README
