**Scope**: Complete specification of Tamarin facts, messages, and state for LoRaMesher DV routing

## 1. Core Facts (Persistent State)

### 1.1 Node Facts

```tamarin
// Node identity and role
Node(node_id: value, role: {honest, intermediate, gateway})
  Represents: Node existence in network
  Persistence: Entire execution
  Cardinality: One per node in topology

Honest(node_id: value)
  Represents: Node is not compromised
  Persistence: Entire execution (or until Compromise rule fires)
  Usage: Guard lemmas; prevent analysis of compromised paths

Compromise(node_id: value)
  Represents: Attacker has gained node's secrets
  Persistence: From first Compromise event onwards
  Semantics: node outputs PSK, RouteTable; attacker can forge messages
```

### 1.2 Cryptographic State

```tamarin
PSK(node_id: value, key: value)
  Represents: Node's pre-shared encryption key
  Persistence: Entire execution
  Cardinality: One per honest node
  Assumption: Established out-of-band before topology start
  Leakage: Node compromise rule outputs this fact

PSKShared(node_i: value, node_j: value, key: value)
  Represents: Shared key between two neighbors (alternative model)
  Persistence: Entire execution
  Cardinality: One per neighbor pair (if used)
  Note: Current model uses single PSK per node (simplified)
```

### 1.3 Hello & Sequence State

```tamarin
HelloSeq(node_id: value, seq: value)
  Represents: Current Hello sequence number for this node
  Persistence: Updated every Hello broadcast
  Type: Monotonically increasing (seq' > seq)
  Range: uint16 (0..65535), modeled as nat

LastHelloSeq(node_id: value, seq: value)
  Represents: Sequence number of last Hello from node_id seen by this node
  Persistence: Updated when route update occurs
  Semantics: Prevents accepting Hello with seq < LastHelloSeq (replay protection)
```

### 1.4 Route Table State

```tamarin
RouteTable(node_id: value, dst: value, metric: value, nextHop: value, seq: value)
  Represents: Learned route to destination
  Persistence: Updated by Route_Update rules; cleared by Route_Timeout
  Fields:
    - node_id: Owner of this route table entry
    - dst: Destination reachable via this entry
    - metric: Number of hops (or cost)
    - nextHop: Immediate neighbor to forward toward dst
    - seq: Sequence number of the Hello that established this route
  Cardinality: At most N entries per node (N = network size)
  Invariant: metric >= 1 (or ∞ for unreachable)

RouteMetricVersion(node_id: value, dst: value, metricVer: value)
  Represents: Version counter for metric updates (for patched protocol)
  Persistence: Incremented on each metric update to same dst
  Semantics: Prevents old metric from being accepted even if seq is same
```

### 1.5 Topology State

```tamarin
Neighbor(node_i: value, node_j: value)
  Represents: Wireless 1-hop neighborhood
  Persistence: Entire execution (static topology assumption)
  Semantics: Determines which nodes can directly hear each other's broadcasts
  Cardinality: Depends on network topology; can be dense (O(n²)) or sparse
```

### 1.6 Attacker Knowledge

```tamarin
AttackerKnows(value: value)
  Represents: Attacker has deduced this value through Dolev-Yao rules
  Persistence: Once known, remains known (monotonic increase)
  Generated by: All Out() facts, Compromise rules

AttackerPSK(node_id: value, key: value)
  Represents: Attacker explicitly learned PSK of node_id
  Persistence: Entire execution
  Semantics: Enables attacker to forge Hello/Data from this node
```

## 2. Messages & Protocol Events

### 2.1 Hello Message (Baseline LoRaMesher)

```tamarin
// Outbound: Node broadcasts Hello
Hello(sender: value, metric: value, seq: value, routing_table: value)
  Fields:
    - sender: Node ID of originator
    - metric: Distance from sender (0 = self, 1 = first rebroadcast, etc.)
    - seq: Sequence number (monotonic per sender)
    - routing_table: Encoded routing info (simplified as single value in model)
  
  Example: Hello(node_1, 0, 42, routing_info_1)
  Representation: Broadcast message (single transmission, multiple receivers)

// Inbound: Neighbor receives Hello
In(Hello(sender, metric, seq, routing_table))
  Semantics: Node receives Hello and can now execute Route_Update rule
  Prerequisite: Receiver must be Neighbor(receiver, sender)
```

### 2.2 Hello Message (Patched Protocol)

```tamarin
Hello_Authenticated(sender: value, metric: value, seq: value, 
                     metric_version: value, hmac_tag: value)
  Additional fields (patched):
    - metric_version: Monotonic counter for metric updates (same seq)
    - hmac_tag: HMAC(sender, metric, seq, metric_version, key=PSK(sender))
  
  Verification rule:
    Receive -> Verify hmac_tag before route update
    If verify fails -> Drop message
    If verify succeeds -> Proceed with Bellman-Ford update
```

### 2.3 Data Message

```tamarin
Data(src: value, dst: value, payload: value, seq_data: value)
  Fields:
    - src: Originating node
    - dst: Final destination
    - payload: Application data (treated as symbolic)
    - seq_data: Data packet sequence (different from Hello seq)
  
  Semantics: Intermediate nodes forward via RouteTable lookup

Encrypt(plaintext: value, key: value, from: value, to: value) → ciphertext
  Functions: enc(plaintext, key) → ciphertext
  Equation: dec(enc(p, k), k) = p
  Assumption: Only holder of key can decrypt
  
  Usage: Attacker cannot read encrypted Data unless:
    - Attacker knows key
    - Attacker is intermediate and decrypts on first hop
```

### 2.4 Internal Control Events (Tamarin Actions)

```tamarin
--[ HelloSent(node: value, metric: value, seq: value) ]->
  Represents: Node has broadcast a Hello message
  Used in: Lemmas to prove reachability and timing

--[ RouteUpdate(node: value, via: value, metric: value, seq: value) ]->
  Represents: Node updated its route table (Bellman-Ford update)
  Fields clarify: Which node, via which neighbor, with what new metric/seq

--[ Forward(forwarder: value, src: value, dst: value, nextHop: value, metric: value) ]->
  Represents: Intermediate node forwarded data packet
  Used in: Path consistency lemmas

--[ RouteMetricAuth(node: value, src: value, metric: value, hmac_valid: bool) ]->
  Represents: HMAC verification result (patched protocol)
  Used in: Metric authenticity lemmas

--[ RouteTimeout(node: value, dst: value) ]->
  Represents: Route declared unreachable (timeout)
  Used in: Route stability lemmas

--[ Compromise(node: value) ]->
  Represents: Attacker has compromised this node
  Semantics: Node outputs PSK + RouteTable; attacker can impersonate
  Used in: Bound attacker capability in lemmas
```

## 3. State Lifecycle & Transitions

### 3.1 Node State Machine (High-Level)

```
┌─────────────┐
│   Startup   │  Node_Init rule
└──────┬──────┘
       │ Generate node_id, PSK
       │ Set HelloSeq = 0
       │ Init RouteTable = empty
       ▼
┌──────────────────────────────────────────┐
│  Active: Wait for Hello Period           │
│  - Listen to In(Hello(...))              │
│  - Store in AttackerKnows               │
└──────────┬─────────────────────────────┬─┘
           │                             │
    [Node's Hello]         [Neighbor's Hello]
           │                             │
           ▼                             ▼
┌─────────────────────┐    ┌──────────────────────────┐
│ Broadcast Own Hello │    │ Route_Update Rule        │
│ HelloSeq++          │    │ Update RouteTable[src]   │
│ --[HelloSent]       │    │ --[RouteUpdate]          │
└─────────────────────┘    └──────────┬───────────────┘
           │                           │
           └───────────────┬───────────┘
                           │
                           ▼
                ┌──────────────────────┐
                │ Data Forwarding Loop │
                │ Forward or Originate │
                │ --[Forward]          │
                └──────────────────────┘
```

### 3.2 RouteTable Entry Lifecycle

```
[Nonexistent]
    │ Route_Update rule with new destination
    ▼
[Exist: metric=X, seq=N, nextHop=Y, metricVer=0]
    │ 
    ├─ Hello_Update (same dst, higher seq) ──→ [metric=X', seq=N+1, metricVer=0]
    │
    ├─ Metric_Update (same seq, new metric) ──→ [metric=X', seq=N, metricVer=V+1]
    │
    └─ Route_Timeout (no update for T) ──────→ [metric=∞, seq=N, stale]
        └──→ Eventually removed from table
```

### 3.3 Node Compromise Event

```
┌─────────────────────────┐
│  Normal Execution       │
│ (Node not in DY model)  │
└─────────────┬───────────┘
              │
              │ Compromise rule fires
              ▼
┌──────────────────────────────────────┐
│  Node Compromised                    │
│  Output: PSK(node, key)              │
│  Output: RouteTable(node, ...)       │
│  Attacker can now:                   │
│  - Forge Hello/Data from this node   │
│  - Decrypt any messages to this node │
│  - Know all its routes               │
└──────────────────────────────────────┘
        │
        │ Lemmas must guard against compromised nodes
        ▼
┌──────────────────────────────────────┐
│  Lemma verification:                 │
│  ∀ n. Honest(n) ∧ ¬Compromise(n)     │
│    ==> Security property holds       │
└──────────────────────────────────────┘
```

## 4. Assertion Rules (Lemma Checkpoints)

### 4.1 Authentication Assertion

```tamarin
// Attacker successfully forged Hello
--[ FakingHello(attacker: value, claimed_src: value, metric: value, seq: value) ]->
  Used in: Lemma_RouteMetricAuth
  Negation: Lemma should prove ¬(FakingHello(...) ∧ Honest(claimed_src))
```

### 4.2 Route Learning Assertion

```tamarin
--[ LearnedRoute(node: value, dst: value, via: value, metric: value, seq: value) ]->
  Used in: Lemma_RouteConsistency
  Semantics: Two nodes should agree on route to same dst
```

### 4.3 Forward Path Assertion

```tamarin
--[ DataOnPath(src: value, dst: value, path: value, timestamp: value) ]->
  Used in: Lemma_RouteCorrectness
  Semantics: Data from src to dst follows a consistent path
```

## 7. Message Timing & Ordering

### 7.1 Relative Ordering Constraints

```
Timeline:
  t1: Node_A broadcasts Hello(A, 0, 100)
  t2: Node_B receives Hello
  t3: Node_B broadcasts Hello(A, 1, 100) ← rebroadcast
  t4: Node_C receives Hello
  
Constraints (Tamarin):
  - No absolute time; only event ordering matters
  - Lemmas can reason about: "If RouteUpdate happens, then HelloSent happened first"
  - Parallelism: Multiple Hellos can be in flight simultaneously
```

### 7.2 Out-of-Order Delivery

```
Tamarin models:
  - Attacker can delay/reorder messages (Dolev-Yao)
  - Cannot prevent delivery (no loss model in baseline)
  - Receiver must handle: seq_new < seq_old (replay detection)
```

## 8. Example Execution Trace

### 8.1 Baseline LoRaMesher (Vulnerable)

```
Rule: Node_Init
  ✓ Node(1, honest), PSK(1, k1), HelloSeq(1, 0)
  ✓ Node(2, honest), PSK(2, k2), HelloSeq(2, 0)
  ✓ Node(3, honest), PSK(3, k3), HelloSeq(3, 0)
  ✓ Neighbor(1,2), Neighbor(2,3)

Rule: Node_A_Hello_Broadcast
  ✓ Out(Hello(1, 0, 1, <info>))
  ✓ HelloSeq(1, 1)

Rule: Node_B_Receive
  ✓ In(Hello(1, 0, 1, <info>))
  ✓ RouteTable(2, 1, 1, 1, 1)  ← 1 hop to reach node 1 via node 1

Rule: Node_C_Receive (via B's rebroadcast)
  ✓ In(Hello(1, 1, 1, <info>))
  ✓ RouteTable(3, 1, 2, 2, 1)  ← 2 hops to reach node 1 via node 2

Rule: ATTACKER_Forge_Hello
  ✓ Out(Hello(1, 0, 1, <fake>))  ← Attacker sends as if from node 1!
  ✓ AttackerKnows(Hello(1, 0, 1, <fake>))

Rule: Node_C_Receive_Fake
  ✓ In(Hello(1, 0, 1, <fake>))
  ✓ RouteTable(3, 1, 1, 3, 1)  ← COMPROMISED: Now routes to node 1 via attacker (node 3)!
  
ATTACK SUCCESS: Route poisoning achieved without authentication
```

### 8.2 Patched LoRaMesher (Secure)

```
Rule: Node_A_Hello_Broadcast_Patched
  ✓ Out(Hello_Auth(1, 0, 1, 0, hmac(concat(...), k1)))
  ✓ HelloSeq(1, 1)

Rule: ATTACKER_Forge_Hello_Patched
  ✗ Out(Hello_Auth(1, 0, 1, 0, <fake_hmac>))
  ✗ Verify fails: hmac ≠ hmac(concat(...), k1)
  ✗ Rule Route_Update_Verify guards: only accepts if HMAC correct
  
ATTACK PREVENTED: Without k1, attacker cannot forge valid Hello
```

## Appendix: State Constraint Summary

### Invariants Expected to Hold (Pre-Verification)

1. **Sequence Monotonicity**: HelloSeq(n, s) ∧ HelloSeq(n, s') ⟹ s < s' (if s' is later)
2. **Metric Ordering**: RouteTable(n, dst, m1, h1, seq1) ∧ RouteTable(n, dst, m2, h2, seq2) with seq1=seq2 ⟹ m1=m2 (unique best path)
3. **Neighborhood Symmetry**: Neighbor(i, j) ⟹ Neighbor(j, i) (optional; can model asymmetric wireless)
4. **Hop Limit**: metric ≤ n-1 (can't exceed network diameter)

### Invariants Expected to BREAK (Vulnerabilities Found)

1. ❌ **No Metric Authenticity**: Attacker can forge any metric value
2. ❌ **No Freshness Binding**: Old seq can be paired with new metric
3. ❌ **No Hop Authentication**: Intermediate can modify metric freely
